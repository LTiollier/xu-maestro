### Pattern 1 — Découvertes de packages/config au moment de l'implémentation

| Story | Surprise |
|---|---|
| 1.3 | `routes/api.php` inexistant dans Laravel 13 |
| 1.3 | `bootstrap/app.php` sans `api:` → 404 sur tous les appels API |

| 1.5 | `@xyflow/react` non installé |
| 1.5 | `steps[]` absent de `WorkflowResource` et des types TS |

Ces découvertes ralentissent le démarrage de chaque story.

### Pattern 2 — Tailwind v4 répété dans 4 stories sur 5

La même information (pas de `tailwind.config.ts`, config dans `globals.css` via `@theme inline`) a été re-documentée dans les CRITICAL STATE de 1.1, 1.2, 1.4 et 1.5. Information critique mais répétitive.

### Pattern 3 — Désynchronisation des responsabilités entre stories

Story 1.1 prétend avoir ajouté les tokens `agent-*`. Story 1.2 les ajoute réellement. Ce genre d'incohérence dans les dev records complique la lecture de l'historique.

## Lessons learned

1. **Les revues de code sont structurelles, pas optionnelles.** Chaque revue a trouvé 2–3 bugs non-triviaux. Sans elles, au moins 3 comportements silencieux auraient atteint l'Épic 2.

2. **Le cycle de vie React est le point faible récurrent.** `AbortController`, `nodeTypes` hors composant, réconciliation state — des patterns à ajouter systématiquement à la checklist de revue.

3. **La préparation des dépendances doit être explicite.** Lister les packages npm/composer à installer avant l'implémentation évite les surprises de démarrage.

4. **Les vérités codebase permanentes méritent un document central.** `project-context.md` permettrait de ne pas répéter Tailwind v4, PHPUnit 12, Laravel 13 bootstrap dans chaque story.
